框架介绍

Ooder 是一个基于注解驱动的全栈开发框架，通过丰富的注解体系实现前后端的强映射关系。平台采用组件化设计理念，通过"干-支-叶"的结构化设计模式，实现了从基础组件到复杂应用场景的完整覆盖。

核心特性

  • 注解驱动开发 - 通过声明式注解配置实现视图结构和行为定义
  • 组件化设计 - "干-支-叶"结构化设计模式，支持复杂业务场景
  • 全栈架构 - 后端注解与前端映射无缝集成
  • 多主题支持 - 内置亮色、暗色、高对比度等多种主题
  • 响应式设计 - 完美适配多种设备，提供一致的用户体验
  • 企业级特性 - 权限管理、数据验证、国际化等企业级功能

架构设计

Ooder 框架采用分层架构设计，主要包括以下几个核心层次：

后端注解层

提供丰富的注解体系，用于定义业务模块、视图组件、服务接口等：

@BusinessModule("客户管理") @Agent("客户列表") @AgentAction("添加客户")

前端映射层

自动将后端注解映射为前端组件，实现前后端一致性：

// 自动生成前端组件 // 基于注解配置渲染界面

组件系统层

提供丰富的UI组件库，支持自定义扩展：

  • 表单视图 - 数据录入和展示
  • 列表视图 - 数据展示和操作
  • 门户视图 - 信息整合展示
  • 树形视图 - 层级结构数据展示

注解体系

Ooder 框架提供完整的注解体系，涵盖业务开发的各个方面：

核心注解

注解名称 用途 示例
@BusinessModule 定义业务模块 @BusinessModule("客户管理")
@Agent 定义业务代理 @Agent("客户列表")
@AgentAction 定义业务操作 @AgentAction("添加客户")
@Access 权限控制 @Access("READ_ONLY")
@AttributeName 属性命名 @AttributeName("客户姓名")

注解使用示例

@BusinessModule( value = "客户管理", desc = "客户信息管理和维护模块", moduleCode = "CRM-CUSTOMER", version = "1.0.0" ) public class CustomerModule { @Agent( name = "客户列表", desc = "客户信息查询和展示" ) @Access("READ_ONLY") public void listCustomers() { // 业务逻辑实现 } @AgentAction( name = "添加客户", desc = "新增客户信息" ) @Access("WRITE") public void addCustomer(@AttributeName("客户姓名") String name) { // 添加客户逻辑 } }

组件系统

Ooder 框架提供丰富的组件系统，支持各种业务场景：

视图组件

  • 表单视图 - 数据录入和展示，支持复杂表单验证
  • 列表视图 - 数据展示和操作，支持分页、排序、筛选
  • 门户视图 - 信息整合展示，支持拖拽布局
  • 树形视图 - 层级结构数据展示，支持展开/折叠
  • 标签页视图 - 复杂界面组织，支持动态标签页
  • 分组视图 - 相关元素归类，支持嵌套分组

服务组件

  • BAR服务 - 工具栏和底部栏操作
  • 通用视图服务 - 视图业务逻辑处理
  • 导航服务 - 视图间跳转处理
  • 模块跳转服务 - 模块间跳转处理

通讯组件

  • APIEvent - 前后端交互事件处理
  • MQTT - 轻量级消息传输
  • 按钮事件 - 用户界面按钮交互
  • 自定义动作 - 组件事件响应

快速开始

按照以下步骤快速开始使用 Ooder 框架：

1. 环境要求

  • Java 8 或更高版本
  • Maven 3.6 或更高版本
  • 支持 Servlet 3.0+ 的 Web 容器

2. 添加依赖

在 Maven 项目的 pom.xml 中添加依赖：

<dependency> <groupId>net.ooder</groupId> <artifactId>ooder-annotation</artifactId> <version>1.0.1</version> </dependency>

3. 创建业务模块

使用注解定义第一个业务模块：

@BusinessModule( value = "示例模块", desc = "Ooder 框架示例模块", moduleCode = "EXAMPLE", version = "1.0.0" ) public class ExampleModule { @Agent( name = "示例列表", desc = "示例数据展示" ) public void listExamples() { // 业务逻辑实现 } @AgentAction( name = "添加示例", desc = "新增示例数据" ) public void addExample() { // 添加示例逻辑 } }

4. 配置前端

在 resource/static/ood 目录下配置前端资源，框架会自动映射后端注解。

最佳实践

遵循以下最佳实践，充分发挥 Ooder 框架的优势：

设计原则

  • 单一职责原则 - 每个组件应有明确的职责和用途
  • 可复用性原则 - 设计通用组件，提高代码复用率
  • 可维护性原则 - 保持代码结构清晰，易于理解和修改

实现规范

  • 注解使用规范 - 正确使用各类注解，保持一致性
  • 服务设计规范 - 遵循服务规范，确保Web可访问性
  • 参数设计规范 - 遵循参数最小化原则
  • 异常处理规范 - 提供完善的异常处理机制

性能优化

  • 合理使用缓存机制
  • 优化数据库查询
  • 减少不必要的组件渲染
  • 使用异步加载技术

API 参考

核心 API

  • 事件管理器 - 管理组件间事件通信
  • 主题管理器 - 动态切换主题样式

扩展 API

  • 自定义注解 - 支持定义业务特定注解
  • 插件系统 - 支持功能模块插件化
  • 国际化支持 - 多语言资源管理
  • 权限控制 - 细粒度权限管理
