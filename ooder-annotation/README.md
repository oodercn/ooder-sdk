# ooder注解驱动框架技术文档

## 概述

ooder平台是一个基于注解驱动的前端框架，通过丰富的注解体系实现前后端的强映射关系。平台采用组件化设计理念，通过"干-支-叶"的结构化设计模式，实现了从基础组件到复杂应用场景的完整覆盖。

## 核心概念

### 注解驱动架构
通过声明式注解配置实现视图结构和行为定义，将配置与业务逻辑分离，提高代码的可维护性和可读性。

### 组件化设计
将复杂功能拆分为可复用的组件模块，通过组合不同组件实现复杂业务场景。

### "干-支-叶"设计模式
ooder平台的核心组件体系遵循"干-支-叶"结构化设计模式：
- **干**：核心注解类
- **支**：组件体系子类
- **叶**：具体应用场景实现

### 包结构调整（版本2.0）
为配合ooderAgent开源版本，版本2.0对包结构进行了调整：
- **旧包路径**：net.ooder.esd.annotation
- **新包路径**：net.ooder.annotation

新的包结构更加简洁，便于与ooderAgent等其他模块集成。

## 主要组件类型

### 视图组件
1. **表单视图** - 数据录入和展示
2. **列表视图** - 数据展示和操作
3. **门户视图** - 信息整合展示
4. **树形视图** - 层级结构数据展示
5. **标签页视图** - 复杂界面组织
6. **分组视图** - 相关元素归类

### 服务组件
1. **BAR服务** - 工具栏和底部栏操作
2. **通用视图服务** - 视图业务逻辑处理
3. **导航服务** - 视图间跳转处理
4. **模块跳转服务** - 模块间跳转处理

### 通讯组件
1. **APIEvent** - 前后端交互事件处理
2. **MQTT** - 轻量级消息传输
3. **按钮事件** - 用户界面按钮交互
4. **自定义动作** - 组件事件响应

### CSS样式组件（版本2.0新增）
1. **基础样式注解** - 原子级CSS属性配置
2. **组件样式注解** - 针对特定组件的结构化样式
3. **预设样式** - 基于现代UI框架的样式方案
4. **CSS枚举类型** - 类型安全的CSS属性值

## 最佳实践

### 设计原则
1. **单一职责原则** - 每个组件应有明确的职责和用途
2. **可复用性原则** - 设计通用组件，提高代码复用率
3. **可维护性原则** - 保持代码结构清晰，易于理解和修改
